## Overview

Starting in Phoenix v14, experiments run inside the Phoenix server as a background daemon rather than in the calling Python process. When you call `run_experiment`, Phoenix enqueues the work server-side and your script can exit — the experiment keeps running and its results appear in the UI as they complete.

Key properties of the background runner:

- **Process-independent** — close your terminal or notebook and the experiment continues.
- **Crash recovery** — if the server restarts, in-progress experiments resume automatically from where they left off.
- **Multi-replica safe** — in high-availability PostgreSQL deployments, one replica owns each experiment job. Heartbeats detect crashed replicas and a periodic orphan scan re-claims their work within ~10 minutes.
- **Fair scheduling** — the server runs up to 1,000 concurrent task slots shared across all active experiments, dispatched round-robin so no single experiment monopolizes capacity.

## Running an experiment

The `run_experiment` call is unchanged from previous versions. It submits the experiment to the server and returns once results are recorded.

<CodeGroup>
```python Sync
from phoenix.client import Client
from phoenix.client.experiments import run_experiment

client = Client()
dataset = client.datasets.get_dataset(dataset="my-dataset")

def my_task(input):
    # call your LLM here
    return f"Hello {input['name']}"

experiment = run_experiment(
    dataset=dataset,
    task=my_task,
    experiment_name="my-experiment",
)
```

```python Async
from phoenix.client import AsyncClient
from phoenix.client.experiments import async_run_experiment

client = AsyncClient()
dataset = await client.datasets.get_dataset(dataset="my-dataset")

async def my_task(input):
    return f"Hello {input['name']}"

experiment = await async_run_experiment(
    dataset=dataset,
    task=my_task,
    experiment_name="my-experiment",
)
```
</CodeGroup>

## Resuming an experiment

If an experiment was interrupted — because the server restarted, a circuit breaker tripped, or you stopped it manually — you can resume it with `resume_experiment`. The runner queries the database for incomplete runs and only executes the missing work; completed runs are not re-executed.

<CodeGroup>
```python Sync
from phoenix.client.experiments import resume_experiment

def my_task(input):
    return f"Hello {input['name']}"

resume_experiment(
    experiment_id="UGhvZW5peEV4cGVyaW1lbnQ6MQ==",
    task=my_task,
)
```

```python Async
from phoenix.client.experiments import async_resume_experiment

async def my_task(input):
    return f"Hello {input['name']}"

await async_resume_experiment(
    experiment_id="UGhvZW5peEV4cGVyaW1lbnQ6MQ==",
    task=my_task,
)
```
</CodeGroup>

You can find the experiment ID in the Phoenix UI or from the `RanExperiment` object returned by `run_experiment`.

<Note>
Multi-output evaluators (evaluators that return a list of results) are not supported for resume operations. Each evaluator must produce a single result.
</Note>

### Resuming with evaluators

Pass `evaluators` to run (or re-run) evaluations on all completed task runs after resuming:

```python
from phoenix.client.experiments import resume_experiment

def correctness(output, expected):
    return 1.0 if output.strip() == expected.strip() else 0.0

resume_experiment(
    experiment_id="UGhvZW5peEV4cGVyaW1lbnQ6MQ==",
    task=my_task,
    evaluators={"correctness": correctness},
)
```

## Running evaluations on a completed experiment

To add new evaluators to an already-completed experiment without re-running the task, use `get_experiment` + `evaluate_experiment`:

```python
from phoenix.client.experiments import get_experiment, evaluate_experiment

experiment = get_experiment(experiment_id="UGhvZW5peEV4cGVyaW1lbnQ6MQ==")

def my_new_evaluator(output, expected):
    return {"score": 1.0 if output == expected else 0.0}

evaluate_experiment(
    experiment=experiment,
    evaluators=[my_new_evaluator],
)
```

## Ephemeral experiments

Experiments created from the **Playground** are marked ephemeral. Ephemeral experiments:

- Stop automatically when you navigate away from the Playground page (the server-side subscription closes and the runner stops the job).
- Are permanently deleted by the sweeper daemon after **24 hours** of inactivity.

Experiments created via `run_experiment` from the Python client are **never** ephemeral. They run until complete (or until you stop them from the UI) and are never auto-deleted.

## Multi-replica behavior (HA deployments)

When Phoenix runs with multiple replicas backed by PostgreSQL:

- Each replica has a unique ID. When it claims an experiment, it records the claim timestamp in the database.
- A **heartbeat loop** refreshes the claim every 5 minutes to signal the replica is still alive.
- A periodic **orphan scan** (every ~10 minutes) looks for experiments with stale claims — indicating the owning replica crashed — and re-claims them.
- Atomic `UPDATE ... WHERE claimed_at IS NULL` statements prevent two replicas from claiming the same experiment simultaneously.

The net effect: if a replica running your experiment crashes, another replica picks it up within approximately 10 minutes and resumes from the last recorded state.

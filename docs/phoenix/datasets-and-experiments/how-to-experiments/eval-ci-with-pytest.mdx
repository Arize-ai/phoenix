title: "Eval CI with pytest"
description: "Run LLM evals as pytest tests and record them as Phoenix experiments"
---

The Phoenix pytest plugin lets you write LLM evaluations as ordinary pytest
tests. Each marked test is recorded as a run in a Phoenix experiment, and pytest's
exit code becomes your CI gate. A test suite maps to a dataset, each test case maps
to a dataset example, and the assertion outcome is recorded as a reserved `pass`
annotation.

## Installation

The plugin ships with `arize-phoenix-client` and is activated by the `pytest` extra.
Once installed, it registers itself through pytest's plugin entry point — no
configuration is required.

```bash
pip install "arize-phoenix-client[pytest]" pytest
```

If your tests use `phoenix.evals` evaluators (via `@pytest.mark.phoenix(evaluators=...)`
or `px.evaluate(...)`), also install the `evals` extra:

```bash
pip install "arize-phoenix-client[pytest,evals]" pytest
```

## Marking tests

Mark any test with `@pytest.mark.phoenix` to record it. Combine it with
`pytest.parametrize` to turn each parameter set into a dataset example, and use
the module-level `px.*` helpers to record output and scores.

```python
import pytest

import phoenix.client.pytest_plugin as px


@pytest.mark.phoenix(dataset="qa-suite")
@pytest.mark.parametrize(
    "question,expected",
    [("What is 2+2?", "4"), ("Capital of France?", "Paris")],
    ids=["arithmetic", "geography"],
)
def test_answers(question, expected):
    result = my_app(question)
    px.log_output(result)
    px.log_evaluation(name="exact_match", score=1.0 if result == expected else 0.0)
    assert result == expected
```

### The marker

`@pytest.mark.phoenix(dataset=None, evaluators=None)` accepts:

- `dataset` — the dataset (and experiment) name. Defaults to the test module's
  name when omitted. A `phoenix_dataset` value in `pytest.ini` overrides every
  marker in the session and collapses the whole run into one experiment.
- `evaluators` — one or more `phoenix.evals` evaluators run automatically against
  each case, in addition to the assertion-derived `pass` annotation. This is the
  "one task over a dataset with a uniform set of evaluators" shape.
- `repetitions` — run the test multiple times (see [Repetitions](#repetitions)).

Stable identity comes from the `parametrize` ids: re-running the same suite maps
each case to the same dataset example, so runs accumulate as experiments over the
same fixtures rather than creating duplicates.

### Recording helpers

Import the plugin module as `px` and call its helpers from inside a marked test:

- `px.log_output(output)` — record the system's output for the run. Capture is
  explicit because pytest warns on non-`None` test returns.
- `px.log_evaluation(name=..., score=..., label=..., explanation=...)` — attach an
  extra annotation to the run, keyed by `name`. The annotation is recorded
  alongside the run; calling this does not by itself fail the test (only a failed
  assertion does).
- `px.evaluate(evaluator, **inputs)` — run a `phoenix.evals` evaluator inline,
  record its score(s) on the run, and return the raw result so you can assert on
  it. An assertion on the result feeds the `pass` annotation, so an inline
  evaluation can gate the individual test.

The helpers read a per-test context set by the plugin, so they take no fixture
parameter and also work in `unittest.TestCase` methods (which receive no fixtures).
Calling a helper outside a marked test raises an error.

### Repetitions

Because LLM outputs vary between runs, you can repeat each test to reduce noise.
Per-test `repetitions=` on the marker takes precedence over the session-wide
`PHOENIX_TEST_REPETITIONS` env var, which takes precedence over the default of `1`.

```python
@pytest.mark.phoenix(dataset="qa-suite", repetitions=3)
def test_flaky_answer():
    ...
```

Repetitions are expanded into real pytest items, so they are visible to `-k`,
`pytest-xdist`, and your IDE. All repetitions of one case share the same dataset
example and are recorded as distinct runs.

## Offline mode

Iterate locally without sending anything to Phoenix by setting either env var.
In offline mode the suite runs normally and prints how many tests ran without
recording.

```bash
PHOENIX_TEST_DRY_RUN=1 pytest    # collect + run, skip all network calls
PHOENIX_TEST_TRACKING=0 pytest   # disable recording entirely
```

## The `PHOENIX_TEST_*` env contract

The plugin is configured entirely through environment variables, so the same test
suite behaves differently in local development and CI without code changes.
Booleans accept `1/true/yes/on` (truthy) and `0/false/no/off`/empty (falsy).

| Variable | Default | Meaning |
| --- | --- | --- |
| `PHOENIX_TEST_TRACKING` | `true` | Master switch for recording. When false, the suite runs offline. |
| `PHOENIX_TEST_DRY_RUN` | `false` | Run and collect, but make no network calls (also offline). |
| `PHOENIX_TEST_REPETITIONS` | `1` | Default repetition count for marked tests; must be an integer `>= 1`. |
| `PHOENIX_TEST_REPO_INFO` | `true` | Collect git provenance into the experiment's `repo_info` metadata. |

<Note>
These names and their truthiness semantics are a cross-ecosystem contract shared
verbatim with the TypeScript test runner. A malformed value (for example a
non-integer `PHOENIX_TEST_REPETITIONS`) is a hard error so CI never silently runs
misconfigured.
</Note>

Connection to your Phoenix deployment uses the standard client variables —
`PHOENIX_COLLECTOR_ENDPOINT`, `PHOENIX_API_KEY`, and `PHOENIX_CLIENT_HEADERS`.

## CI gating

pytest's exit code is your CI gate, with no configuration required: a failing
assertion records `pass=False` for that run and fails the pytest item, exactly
like a normal test, so the job fails. Uploads to Phoenix are best-effort and never
fail a test on their own — a network problem degrades to a warning rather than a
red build.

The contract a CI consumer depends on is small and stable: **pytest's exit code
and the printed terminal summary**. Everything else (experiment IDs, payloads) is
an implementation detail you may consume but should not treat as frozen.

## The `repo_info` convention

When `PHOENIX_TEST_REPO_INFO` is enabled (the default), the plugin records git
provenance under the reserved `repo_info` key of the experiment's `metadata`:

```json
{
  "commit": "…",
  "branch": "…",
  "dirty": false,
  "author_name": "…",
  "author_email": "…",
  "commit_message": "…",
  "commit_time": "…",
  "remote_url": "…",
  "tag": "…"
}
```

Collection degrades gracefully: outside a git repository the block is simply
absent. Credentials in `remote_url` are stripped. `repo_info` is reserved for
runner-collected metadata — a runner overwrites any user-supplied `repo_info`.

## GitHub Actions recipe

The following workflow installs the plugin, runs your eval suite, and gates the job
on the pytest exit code. Copy it to `.github/workflows/eval-ci.yml`. A runnable copy
plus an example test suite lives in the
[client examples directory](https://github.com/Arize-ai/phoenix/tree/main/packages/phoenix-client/examples/pytest_plugin).

```yaml
name: eval-ci

on:
  pull_request:

jobs:
  evals:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      # The pytest extra activates the plugin entry point.
      - name: Install dependencies
        run: pip install "arize-phoenix-client[pytest]" pytest

      - name: Run eval suite
        env:
          PHOENIX_COLLECTOR_ENDPOINT: ${{ secrets.PHOENIX_COLLECTOR_ENDPOINT }}
          PHOENIX_API_KEY: ${{ secrets.PHOENIX_API_KEY }}
        run: pytest tests/evals
```

### Optional: post the summary as a PR comment

The example workflow includes a commented-out step that captures the terminal
summary and upserts it as a single PR comment, using a hidden marker to update its
own previous comment instead of adding a duplicate on every run. It is an example
consumer of the printed summary — adapt it to your needs rather than treating it
as a maintained Phoenix artifact.

<Tip>
The terminal summary and pytest exit code are the only stable contracts. Build
custom CI integrations against those, not against internal plugin behavior.
</Tip>

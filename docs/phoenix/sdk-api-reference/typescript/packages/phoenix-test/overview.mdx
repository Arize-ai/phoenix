---
title: "Overview"
description: "Run dataset-backed Phoenix evaluations as Vitest or Jest tests"
---

`@arizeai/phoenix-test` lets you write evaluations as ordinary Vitest or
Jest tests. Each `describe()` block becomes a Phoenix dataset and a new
experiment; each `test()` becomes a dataset example plus a recorded
experiment run; the assertion outcome is captured as a `pass` boolean
annotation. Anything you log via `logOutput()` / `logAnnotation()` /
`wrapEvaluator()` lands on the run.

Tracing is provided by `@arizeai/phoenix-otel` (OpenInference). LLM and
agent calls instrumented with OpenInference appear as child spans of
each test's task span.

## Install

```bash
npm install -D @arizeai/phoenix-test vitest dotenv
# or, for jest:
npm install -D @arizeai/phoenix-test jest dotenv
```

## Minimal Example

```ts
import * as px from "@arizeai/phoenix-test";
import { expect } from "vitest";

px.describe("generate sql demo", () => {
  px.test(
    "generates select all",
    {
      input: { userQuery: "Get all users from the customers table" },
      expected: { sql: "SELECT * FROM customers;" },
    },
    async ({ input, expected }) => {
      const sql = await myApp(input.userQuery);
      px.logOutput({ sql });
      expect(sql).toEqual(expected?.sql);
    },
  );
});
```

## Module Map

| Import | Purpose |
|--------|---------|
| `@arizeai/phoenix-test` | Default Vitest entrypoint (`describe`, `test`, `it`, `logOutput`, `logAnnotation`, `wrapEvaluator`) |
| `@arizeai/phoenix-test/vitest` | Explicit Vitest entrypoint |
| `@arizeai/phoenix-test/vitest/reporter` | Vitest reporter that prints a Phoenix-flavored summary at the end of the run |
| `@arizeai/phoenix-test/jest` | Same API surface, wired to Jest globals |
| `@arizeai/phoenix-test/jest/reporter` | Jest reporter |

## Phoenix Terminology

The public API uses Phoenix terms end-to-end. There is no LangSmith
naming layer — what you read in this package matches what shows up in
the Phoenix UI and the REST API.

| `phoenix-test` field        | Phoenix concept                          |
| --------------------------- | ---------------------------------------- |
| `input`                     | `Example.input`                          |
| `expected`                  | `Example.output` (reference)             |
| `metadata`                  | `Example.metadata`                       |
| `id`                        | `Example.id` (stable upsert id)          |
| `logOutput(value)`          | `ExperimentRun.output`                   |
| `Annotation.name`           | `ExperimentEvaluation.name`              |
| `Annotation.score`          | `ExperimentEvaluationResult.score`       |
| `Annotation.label`          | `ExperimentEvaluationResult.label`       |
| `Annotation.explanation`    | `ExperimentEvaluationResult.explanation` |
| `Annotation.annotatorKind`  | `annotator_kind` (`LLM` / `CODE` / `HUMAN`) |

## Configuration

`@arizeai/phoenix-test` reuses `@arizeai/phoenix-client` and
`@arizeai/phoenix-otel`, so configuration goes through the standard
Phoenix env vars.

| Variable | Purpose |
| --- | --- |
| `PHOENIX_HOST` / `PHOENIX_COLLECTOR_ENDPOINT` | Phoenix base URL |
| `PHOENIX_API_KEY` | Bearer token for Phoenix |
| `PHOENIX_PROJECT_NAME` | Override project name for the experiment's traces |
| `PHOENIX_TEST_TRACKING=false` | Disable sync to Phoenix for the current run |
| `PHOENIX_TEST_DRY_RUN=true` | Alias of `PHOENIX_TEST_TRACKING=false` |
| `PHOENIX_TEST_REPETITIONS` | Default number of times to run each test |

`describe()` also accepts `repetitions` and `dryRun` on its config object,
and `test()` accepts `repetitions` and `dryRun` on its params — see
[Vitest](./vitest) / [Jest](./jest).

## Repetitions

Run a test (or a whole suite) more than once to measure non-determinism.
Each repetition is a separate experiment run against the same dataset
example, carrying its own `repetition_number`, so the Phoenix compare view
lines them up. Resolution order: per-test `repetitions` → suite
`repetitions` → `PHOENIX_TEST_REPETITIONS` → `1`.

## Dry-Run Mode

Dry-run executes test bodies (and tracing, when a tracer is attached) but
creates no dataset, experiment, run, or annotations in Phoenix. The
reporter still prints a local summary.

- **Whole process** — `PHOENIX_TEST_TRACKING=false` or
  `PHOENIX_TEST_DRY_RUN=true` (or omit the standard Phoenix env vars).
- **One suite** — `describe(name, fn, { dryRun: true })`.
- **One test** — `test(name, { input, dryRun: true }, fn)`; that case runs
  as an ordinary local test, with no dataset example and nothing uploaded,
  even when the rest of the suite syncs.

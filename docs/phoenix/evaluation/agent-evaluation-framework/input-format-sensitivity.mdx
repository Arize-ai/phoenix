When an LLM judge evaluates whether an agent made the right tool call, it needs to understand three things: what the user asked for, what tools were available, and what the agent actually did. But **how** you present that information to the judge matters — and the effect scales with the complexity of the tool schemas involved.

We benchmarked two formatting approaches — raw JSON dumps versus structured human-readable text — across three judge models using the Berkeley Function-Calling Leaderboard's real-world API dataset. The results show that on complex schemas (the kind you'd encounter in production), human-readable formatting improves GPT-4o's accuracy by over 5 percentage points and reduces false negatives by 15%. But the effect is model-dependent and comes with tradeoffs — smaller models can lose structural error detection in the translation.

---

## The Experiment

A controlled benchmark isolating the effect of input formatting on LLM judge accuracy. One variable changed, everything else held constant.

### Dataset

**Source:** [Berkeley Function-Calling Leaderboard (BFCL) v3](https://gorilla.cs.berkeley.edu/leaderboard.html) `live_multiple` split — community-contributed real-world API schemas from services like Slack, Uber, food delivery apps, and smart home platforms. This is a peer-reviewed benchmark (ICML 2025) and the de facto standard for function-calling evaluation.

**Why this dataset:** The `live_multiple` split contains genuinely complex schemas — tools averaging 3.8 parameters with up to 21 parameters per tool, enum constraints in 62% of tools, and up to 37 candidate tools per example. This complexity is representative of production agent systems, where the format sensitivity effect actually manifests.

**Construction:** From 1,053 BFCL examples, we filtered to 200 with sufficient complexity (≥3 total parameters across tools), biased toward more complex schemas. Each was paired with a correct invocation (from BFCL ground truth) and an incorrect invocation (generated via systematic perturbation), yielding 400 balanced evaluation instances.

**Perturbation types for incorrect examples:**

| Error Type | Count | How Generated |
|---|---|---|
| Wrong tool selected | 50 | Substituted a different tool from the available set |
| Hallucinated parameter | 50 | Added a parameter not in the tool schema |
| Missing required parameter | 50 | Removed a required parameter from the invocation |
| Wrong argument value | 50 | Corrupted an argument value (negated, reversed, etc.) |

**Schema complexity distribution:**

| Bucket | Examples | Description |
|---|---|---|
| Complex (16+ total params) | 264 | Multi-tool examples with rich API schemas |
| Medium (7–15 total params) | 104 | Moderate schemas with several parameters |
| Simple (1–6 total params) | 32 | Minimal schemas |

### The Experimental Variable

**Two formatting conditions for the same data:**

<Tabs>
  <Tab title="Raw JSON (Condition A)">
    Tool schemas and invocations presented as raw JSON — the native format of most tool-calling APIs:

    ```json
    [{"name": "ChaDri.change_drink", "description":
    "Changes the properties of an existing drink order.",
    "parameters": {"type": "dict", "required":
    ["drink_id", "new_preferences"], "properties":
    {"drink_id": {"type": "string", "description":
    "Unique identifier for the drink."}, "new_preferences":
    {"type": "dict", "properties": {"size": {"type":
    "string", "enum": ["small", "medium", "large"]},
    "milk_type": {"type": "string", "enum": ["whole",
    "skim", "oat", "almond", "coconut"]},
    "sweetness_level": {"type": "string", "enum":
    ["none", "light", "regular", "extra"]},
    "temperature": {"type": "string", "enum":
    ["iced", "warm", "hot", "boiling hot"]}}}}}}]
    ```
  </Tab>
  <Tab title="Human-Readable (Condition B)">
    The same information transformed into structured text:

    ```
    ChaDri.change_drink:
      Description: Changes the properties of an existing
                   drink order.
      Parameters:
        - drink_id (required, string): Unique identifier
          for the drink.
        - new_preferences (required, dict): Drink preferences
            - size: [allowed: small, medium, large]
            - milk_type: [allowed: whole, skim, oat,
              almond, coconut]
            - sweetness_level: [allowed: none, light,
              regular, extra]
            - temperature: [allowed: iced, warm, hot,
              boiling hot]
    ```
  </Tab>
</Tabs>

**Controls held constant:** Same judge prompt template, same temperature (0.0), same evaluation criteria, same 400 examples. Only the formatting of `available_tools` and `tool_invocation` fields changed between conditions.

**Judge models tested:** GPT-4o, GPT-4o-mini, and Claude Sonnet 4.

### Results

| Model | Format | Accuracy | Macro F1 | P (correct) | R (correct) | P (incorrect) | R (incorrect) | Cohen's κ |
|---|---|---|---|---|---|---|---|---|
| GPT-4o | Raw JSON | 80.0% | 0.794 | 0.955 | 0.630 | 0.724 | 0.970 | 0.600 |
| GPT-4o | Human-Readable | **85.2%** | **0.852** | 0.917 | **0.775** | **0.805** | 0.930 | **0.705** |
| GPT-4o-mini | Raw JSON | **72.8%** | **0.713** | 0.910 | 0.505 | 0.657 | 0.950 | **0.455** |
| GPT-4o-mini | Human-Readable | 69.8% | 0.687 | 0.811 | 0.515 | 0.645 | 0.880 | 0.395 |
| Claude Sonnet 4 | Raw JSON | 71.2% | 0.689 | 0.967 | 0.440 | 0.638 | 0.985 | 0.425 |
| Claude Sonnet 4 | Human-Readable | 71.5% | 0.692 | 0.978 | 0.440 | 0.639 | 0.990 | 0.430 |

**Format effect (Human-Readable − Raw JSON):**

| Model | Δ Accuracy | Δ Macro F1 | Δ Cohen's κ |
|---|---|---|---|
| GPT-4o | **+5.2%** | **+0.058** | **+0.105** |
| GPT-4o-mini | −3.0% | −0.026 | −0.060 |
| Claude Sonnet 4 | +0.2% | +0.002 | +0.005 |

## What the Data Shows

### Finding 1: Human-readable formatting improves GPT-4o accuracy by 5.2%

GPT-4o — the strongest model tested — showed a clear, directional improvement with human-readable formatting: +5.2% accuracy and +0.105 Cohen's κ. This is a meaningful effect on 400 examples from a public benchmark with real-world API schemas.

The improvement came almost entirely from **reducing false negatives on correct invocations** — the model's recall on correct examples jumped from 63.0% to 77.5% (+14.5 percentage points). With raw JSON, GPT-4o wrongly rejected 37% of valid tool calls. Human-readable formatting cut that to 22.5%.

### Finding 2: The improvement scales with schema complexity

Breaking the results down by schema complexity reveals the mechanism:

| Complexity | GPT-4o Raw | GPT-4o HR | Δ |
|---|---|---|---|
| Complex (16+ params) | 82% | 89% | **+8%** |
| Medium (7–15 params) | 73% | 76% | +3% |
| Simple (1–6 params) | 88% | 81% | −6% |

The effect is strongest on complex schemas (+8%) and actually reverses on simple schemas (−6%). This makes intuitive sense: for a 2-parameter math function, raw JSON is trivially parseable and formatting adds no value. For a 21-parameter Slack API with nested objects and enum constraints, human-readable formatting eliminates substantial parsing burden.

<Info>
  This complexity dependence explains why format sensitivity benchmarks on toy datasets (simple math functions with 2–3 parameters) often show no effect. The BFCL `exec_multiple` split, for example, averages just 1.9 parameters per tool — at that level of simplicity, formatting is irrelevant. The effect only emerges with production-representative schemas.
</Info>

### Finding 3: The tradeoff — structural error detection

Human-readable formatting isn't free. GPT-4o's ability to detect **hallucinated parameters** dropped from 98% to 88% with human-readable formatting:

| Error Type | GPT-4o Raw | GPT-4o HR | Δ |
|---|---|---|---|
| Correct invocations | 63% | **78%** | **+15%** |
| Wrong tool selected | 98% | 96% | −2% |
| Missing required param | 92% | 90% | −2% |
| Hallucinated parameter | **98%** | 88% | **−10%** |
| Wrong argument value | 100% | 98% | −2% |

The same pattern appeared more dramatically in GPT-4o-mini, where hallucinated parameter detection dropped from 94% to 78% (−16%).

**Why this happens:** With raw JSON, the model can perform a direct structural comparison — every field in the invocation must have a corresponding key in the `properties` object. With human-readable format, the parameter list is a natural-language description, and the model must semantically reason about whether an extra parameter like `verbose=True` was listed. This semantic reasoning is more error-prone, especially for smaller models.

<Warning>
  If your evaluation pipeline primarily needs to catch **structural errors** (hallucinated parameters, extra fields, schema violations), raw JSON may serve you better — especially with smaller models. The 5% overall accuracy gain from human-readable formatting comes from reducing false negatives on correct invocations, which may or may not be your primary concern.
</Warning>

### Finding 4: Claude Sonnet 4 is extremely conservative regardless of format

Claude Sonnet 4 showed essentially no format sensitivity (+0.2%). But this masks an interesting pattern: the model had only 44% recall on correct invocations in both conditions — it rejected over half of valid tool calls. Its error detection was near-perfect (98–100% across all error types), but it was so conservative that format made no difference.

This suggests that Claude's evaluation approach is fundamentally different — it may be applying stricter criteria than the ground truth labels warrant, or interpreting ambiguity in the BFCL data more cautiously. For Claude as a judge, the bottleneck isn't format — it's calibration.

## Why Format Matters: The Mechanism

LLMs are trained predominantly on natural language. When presented with nested JSON schemas containing type annotations, enum constraints, and required field lists, the model must perform two tasks simultaneously: **parse the structure** and **evaluate the content**.

Human-readable formatting separates these tasks. Consider what the judge needs to verify for a single tool invocation:

1. Was the correct tool selected from the available set?
2. Are all required parameters present?
3. Are there any extra parameters not in the schema?
4. Do argument values match the user's intent?
5. Are enum values within the allowed set?

With raw JSON, the judge must navigate nested `properties` objects, cross-reference a `required` array, and parse `enum` arrays — all while holding the user's query in context. With human-readable formatting, each parameter is presented with its requirement status, type, and constraints inline.

The complexity-dependent effect confirms this: simple schemas impose minimal parsing burden in either format, while complex schemas create a large gap between the parsing cost of JSON and the directness of human-readable text.

## Practical Implications

### Test format as part of meta-evaluation

Input format is a variable that should be included in your meta-evaluation process (see [Meta-Evaluation: How to Trust Your Evaluators](/docs/phoenix/evaluation/agent-evaluation-framework/meta-evaluation)). When measuring your evaluator's alignment with human labels, test both raw and formatted inputs. The direction and magnitude of the difference depends on your model, your schema complexity, and your error distribution.

### Match format to your priorities

If your primary concern is **catching structural errors** (hallucinated parameters, schema violations): consider raw JSON, especially with smaller models. If your primary concern is **reducing false negatives** (correctly recognizing valid tool calls): use human-readable formatting. For most production pipelines, false negatives are more costly — they generate phantom alerts that waste engineering time — so human-readable formatting is the better default.

### Build formatting into your pipeline

A minimal formatting function for tool schemas:

```python
def format_tool_schema(tool: dict) -> str:
    """Transform a JSON tool schema into human-readable text."""
    lines = [f"{tool['name']}:"]
    lines.append(f"  Description: {tool['description']}")
    params = tool.get("parameters", {}).get("properties", {})
    required = set(tool.get("parameters", {}).get("required", []))
    if params:
        lines.append("  Parameters:")
        for name, defn in params.items():
            req = "required" if name in required else "optional"
            ptype = defn.get("type", "any")
            desc = defn.get("description", "")
            line = f"    - {name} ({req}, {ptype}): {desc}"
            if enum := defn.get("enum"):
                line += f" [allowed: {', '.join(str(e) for e in enum)}]"
            lines.append(line)
    return "\n".join(lines)
```

Arize's built-in tool evaluation metrics (tool selection, tool invocation, and tool response handling) accept both JSON schema and human-readable formats, and their evaluation templates are designed to work optimally with structured, readable input — delivering the formatting benefit automatically.

## Reproducing This Benchmark

The complete benchmark is available in the Phoenix repository at `scripts/benchmarks/input_format_sensitivity_benchmark.py`. It:

- Downloads the BFCL v3 `live_multiple` dataset and ground truth directly from HuggingFace
- Filters to complex schemas and generates balanced correct/incorrect examples
- Runs both formatting conditions across configurable judge models
- Computes precision, recall, F1, accuracy, Cohen's κ
- Reports breakdowns by error type and schema complexity
- Supports OpenAI and Anthropic models via async execution

```bash
export OPENAI_API_KEY="your-key"
export ANTHROPIC_API_KEY="your-key"
python scripts/benchmarks/input_format_sensitivity_benchmark.py
```

<Tip>
  The benchmark is designed to be extended. Adjust `n_examples` and `min_params` in `build_dataset()` to control the sample size and complexity threshold. To test your own formatting approach, add a new pair of formatting functions and a third condition to the experiment loop.
</Tip>

## Key Takeaways

1. **Human-readable formatting improves accuracy by 5% on complex schemas.** On real-world API schemas from the BFCL benchmark, GPT-4o gained +5.2% accuracy and +0.105 Cohen's κ with human-readable formatting — primarily by reducing false negatives on valid tool calls.

2. **The effect scales with schema complexity.** Complex schemas (16+ parameters) saw +8% improvement. Simple schemas (1–6 parameters) saw no benefit or slight degradation. Format sensitivity benchmarks must use production-representative schemas to detect the effect.

3. **There's a tradeoff with structural error detection.** Human-readable formatting reduces the judge's ability to catch hallucinated parameters (−10% for GPT-4o, −16% for GPT-4o-mini). For pipelines focused on schema compliance, raw JSON may be preferable.

4. **The effect is model-dependent.** GPT-4o benefits clearly. GPT-4o-mini gets worse. Claude Sonnet 4 is unaffected. Test your specific model — don't assume.

5. **Format should be a meta-evaluation variable.** When validating your evaluator against human labels, test multiple input formats. The gap tells you how much accuracy is format-dependent.

When an LLM judge evaluates whether an agent made the right tool call, it needs to understand three things: what the user asked for, what tools were available, and what the agent actually did. But **how** you present that information to the judge matters — and not always in the direction you'd expect.

We benchmarked two formatting approaches — raw JSON dumps versus structured human-readable text — across three judge models on the Berkeley Function-Calling Leaderboard (BFCL v3) dataset. The results reveal that format sensitivity is real, but its direction depends on the model, the error type being detected, and the complexity of the tool schemas involved.

This isn't a simple "human-readable is always better" story. It's a more useful one: **input format is a variable that affects your evaluation pipeline's accuracy, and you should test it rather than assume.**

---

## The Experiment

We designed a controlled benchmark to isolate the effect of input formatting on LLM judge accuracy. One variable changed, everything else held constant.

### Dataset

**Source:** [Berkeley Function-Calling Leaderboard (BFCL) v3](https://gorilla.cs.berkeley.edu/leaderboard.html) `exec_multiple` split — a peer-reviewed benchmark (ICML 2025) with real tool schemas and ground-truth invocations. Each example provides 2–4 candidate tool definitions and a user query, with the correct tool call established by the benchmark authors.

**Construction:** From the 50 BFCL examples, we generated 100 evaluation instances — 50 correct (using the ground-truth invocation) and 50 incorrect (using systematic perturbations). This gives us a balanced binary classification task where the judge must distinguish correct from incorrect invocations.

**Perturbation types for incorrect examples:**

| Error Type | Count | How Generated |
|---|---|---|
| Wrong tool selected | 13 | Substituted a different tool from the available set |
| Hallucinated parameter | 13 | Added a parameter not in the tool schema |
| Missing required parameter | 12 | Removed a required parameter from the invocation |
| Wrong argument value | 12 | Corrupted an argument value (negated, reversed, etc.) |

### The Experimental Variable

**The single variable: how tool schemas and invocations are formatted for the judge.**

  </Tab>
  <Tab title="Human-Readable (Condition B)">
    The same information transformed into structured text:

    ```

    calc_binomial_probability:
      Description: Calculates the probability of getting

      Parameters:
        - n (required, integer): The number of trials.
        - k (required, integer): The number of successes.
        - p (required, float): The probability of success.

    get_weather_data:
      Description: Fetches weather data from the Open-Meteo API.
      Parameters:
        - coordinates (required, array): The latitude and

    // Tool invocation (function-call notation)
    calc_binomial_probability(n=20, k=5, p=0.167)
    ```
  </Tab>

**Controls held constant:** Same judge prompt template, same temperature (0.0), same evaluation criteria, same 100 examples. Only the formatting of the `available_tools` and `tool_invocation` fields changed between conditions.

**Judge models tested:** GPT-4o, GPT-4o-mini, and Claude Sonnet 4.

## What the Data Shows

### Finding 1: Format sensitivity is real, but direction varies by model

The headline result: formatting the same data differently changed accuracy by up to 7 percentage points on a public benchmark. But the direction wasn't uniform — Claude Sonnet 4 improved slightly with human-readable formatting, while GPT-4o-mini got meaningfully worse. GPT-4o was completely insensitive to format, producing identical results in both conditions.

This means you can't assume a universal "best format" for LLM judges. The optimal presentation depends on which model you're using as the judge — and the only way to know is to test.

### Finding 2: Human-readable format can hide structural errors

The most striking result is GPT-4o-mini's 7% accuracy drop with human-readable formatting. The per-error-type breakdown reveals why:

| Error Type | Raw JSON Accuracy | Human-Readable Accuracy | Δ |
|---|---|---|---|
| Correct invocations | 92% | 88% | −4% |
| Wrong tool selected | 100% | 100% | 0% |
| Missing required param | 100% | 100% | 0% |
| Hallucinated parameter | **100%** | **69%** | **−31%** |
| Wrong argument value | 92% | 83% | −8% |

The damage is concentrated in **hallucinated parameter detection**: GPT-4o-mini caught 100% of hallucinated parameters with raw JSON but only 69% with human-readable formatting. With raw JSON, the model can do a direct structural comparison — every field in the invocation must appear in the `properties` object. With human-readable format, the parameter list is presented as natural-language descriptions, and a weaker model may not realize that an extra parameter like `verbose=True` or `cache_result=True` has no corresponding entry in the schema.

<Warning>
  Human-readable formatting is not universally better. For error types that require **structural comparison** (hallucinated fields, extra parameters), raw JSON can give the judge a clearer signal — especially with smaller or less capable models. Test your specific model on your specific error distribution before committing to a format.
</Warning>

### Finding 3: Stronger models benefit from (or are neutral to) readable formatting

Claude Sonnet 4 showed a small but consistent improvement with human-readable formatting (+1% accuracy, +0.02 κ). Per-example analysis shows this came from one fewer false negative — a correct invocation that raw JSON caused the judge to wrongly reject.

GPT-4o produced identical results in both conditions — not a single example changed outcome. At 94% accuracy, the model is strong enough that format is irrelevant for this task complexity.

The pattern: **stronger models are either neutral to or slightly helped by human-readable formatting, while weaker models can be hurt by it.** This makes sense — a more capable model can extract the same signal regardless of presentation, while a weaker model may lean on structural cues (like JSON key matching) that disappear in prose format.

### Finding 4: Schema complexity modulates the effect

The BFCL `exec_multiple` schemas are relatively simple — most functions have 2–4 parameters with basic types (integer, float, array, string). We'd expect larger formatting effects with more complex real-world schemas that have:

- Deeply nested objects
- Many parameters (8+) with mixed required/optional
- Enum constraints
- Cross-parameter dependencies

For simple schemas, both formats are easy enough for a strong model to parse. As schema complexity grows, the parsing burden of raw JSON increases while human-readable formatting stays constant in cognitive load. This is consistent with the finding that format effects are small on BFCL but have been observed to be larger in internal benchmarks with production-complexity tool schemas.

<Info>
  The BFCL results represent a **lower bound** on format sensitivity. Real-world agent evaluation involves more complex schemas, multi-turn conversation context, and nested tool outputs — all of which increase the gap between raw and formatted presentations. The directional findings here (model-dependent, error-type-dependent) should hold, but the magnitudes are likely larger in practice.
</Info>

## Why Format Matters: The Underlying Mechanism

LLMs are trained predominantly on natural language. When presented with nested JSON schemas containing type annotations, enum constraints, and required field lists, the model must perform two tasks simultaneously: **parse the structure** and **evaluate the content**.

Human-readable formatting reduces the parsing burden. Consider what the judge needs to verify:

1. Was the correct tool selected from the available set?
2. Are all required parameters present?
3. Are there any extra parameters not in the schema?
4. Do argument values match the user's intent?

With raw JSON, the judge navigates nested `properties` objects, cross-references `required` arrays, and parses `enum` values — all while holding the user's query in context. With human-readable formatting, each parameter is presented with its requirement status, type, and constraints inline.

But our benchmark reveals an important nuance: **the parsing burden cuts both ways.** For verification tasks that are inherently structural (like checking whether every field in an invocation exists in the schema), JSON's structure can actually be an advantage. The model can perform a more mechanical comparison on raw JSON, while natural-language formatting forces semantic reasoning that a weaker model may get wrong.

## Practical Implications

### Test format as part of meta-evaluation

Input format is a variable that should be included in your meta-evaluation process (see [Meta-Evaluation: How to Trust Your Evaluators](/docs/phoenix/evaluation/agent-evaluation-framework/meta-evaluation)). When measuring your evaluator's alignment with human labels, test both raw and formatted inputs. The direction and magnitude of the difference will tell you which format to use for your specific model and error distribution.

### Match format to your error distribution

If your evaluation pipeline primarily needs to catch **structural errors** (hallucinated parameters, schema violations, type mismatches), raw JSON may serve you better — especially with smaller models. If you're evaluating **semantic correctness** (did the agent understand the user's intent? are the values reasonable?), human-readable formatting reduces the parsing burden and lets the judge focus on meaning.

For pipelines that need to catch both, consider running the judge twice with different formats and flagging disagreements for human review.

### Build formatting into your pipeline

Regardless of which format performs better, the key insight is that format is not a neutral choice. A minimal formatting function:

```python
def format_tool_schema(tool: dict) -> str:
    """Transform a JSON tool schema into human-readable text."""
    lines = [f"{tool['name']}:"]
    lines.append(f"  Description: {tool['description']}")
    params = tool.get("parameters", {}).get("properties", {})
    required = set(tool.get("parameters", {}).get("required", []))
    if params:
        lines.append("  Parameters:")
        for name, defn in params.items():
            req = "required" if name in required else "optional"
            ptype = defn.get("type", "any")
            desc = defn.get("description", "")
            line = f"    - {name} ({req}, {ptype}): {desc}"
            if enum := defn.get("enum"):
                line += f" [allowed: {', '.join(str(e) for e in enum)}]"
            lines.append(line)
    return "\n".join(lines)
```

For tool invocations, function-call notation is often clearer than JSON:

```python
def format_invocation(invocation: dict) -> str:
    """Transform a JSON tool call into function-call notation."""
    name = invocation["name"]
    args = invocation.get("arguments", {})
    parts = []
    for k, v in args.items():
        if isinstance(v, str):
            parts.append(f'{k}="{v}"')
        elif isinstance(v, list):
            items = ", ".join(
                f'"{i}"' if isinstance(i, str) else str(i) for i in v
            )
            parts.append(f"{k}=[{items}]")
        else:
            parts.append(f"{k}={v}")
    return f"{name}({', '.join(parts)})"
```

### For evaluation platform selection

When evaluating platforms for agent evaluation, ask whether the platform's built-in evaluators transform structured data before passing it to the judge — and whether they've benchmarked the effect. Arize's built-in tool evaluation metrics (tool selection, tool invocation, and tool response handling) accept both JSON schema and human-readable formats, and their evaluation templates are designed to work optimally with structured input.

## Key Takeaways

1. **Format sensitivity is real and measurable.** On the BFCL benchmark, formatting changed accuracy by up to 7 percentage points — enough to matter in production evaluation pipelines.

2. **The direction depends on the model.** Claude Sonnet 4 improved slightly with human-readable formatting. GPT-4o-mini got worse. GPT-4o was unaffected. There is no universal "best format" — you must test.

3. **Different error types respond differently to format.** Human-readable formatting can hurt detection of structural errors (hallucinated parameters) while helping with semantic errors (intent matching). Match your format to your error distribution.

4. **Schema complexity amplifies the effect.** Simple schemas (2–4 parameters) show small format effects. Complex production schemas with nested objects, enums, and many parameters show larger ones.

5. **Format should be a meta-evaluation variable.** When validating your evaluator against human labels, test multiple input formats. The gap tells you how much accuracy is format-dependent — and which format is right for your use case.

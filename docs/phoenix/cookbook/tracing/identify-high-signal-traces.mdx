When running LLM applications in production, you quickly accumulate thousands or millions of traces. Most of them are fine. A high-signal trace is one that teaches you something you didn't already know about how your system fails — and the discipline of production observability comes down to finding those traces efficiently and actually reading them.

This guide covers:

* What separates signal from noise in production traces
* A two-cadence review practice: a daily smoke check and a weekly error-analysis session
* Filtering traces by errors, latency outliers, evaluation scores, and session behavior — from the CLI and programmatically
* Turning what you find into a failure taxonomy that drives your next evals

## Understanding Signal vs. Noise

The most common mistake teams make is treating observability as a dashboard problem. Generic aggregate metrics — an overall "helpfulness" score, a global pass rate — create a false sense of measurement: the number moves and nobody can say what changed for users. Hamel Husain, who has helped dozens of teams ship production AI systems, calls error analysis "the single most valuable activity in AI development," and observes that successful teams spend the majority of their effort looking at data rather than building automated checks.

So the operative question isn't "what should my dashboard show?" It's "which traces should a human read next?" Everything in this guide — filters, sorts, scores, queries — exists to answer that question. Filters find traces; they never replace reading them.

High-signal traces come from five sources:

1. **Hard failures.** Exceptions, tool-call errors, timeouts, malformed outputs. The cheapest signal to query — and the shallowest. An LLM application can be deeply broken without ever throwing an exception.
2. **Soft failures.** Traces where evaluation scores flag a quality problem: a hallucinated answer, an irrelevant retrieval, a failed task.
3. **Outliers.** Traces at the extremes of any metric — latency, token count, tool-call count, response length. Both directions are interesting: the slow tail reveals retry loops and runaway agents; the suspiciously fast tail reveals silent failures like empty retrievals.
4. **Human signals.** Negative feedback, support escalations, sessions where the user rephrases the same question three times. A user retry is an eval score the user computed for you.
5. **Novelty.** Inputs unlike anything you've categorized before. New usage patterns are where your next failure mode is hiding.

One principle ties these together: **prioritize the failures that happen most, not every failure that's possible.** Review time and evals aren't free. The goal is a ranked list of real, frequent problems — not exhaustive coverage.

## Two Cadences of Trace Review

Production trace review works on two clocks:

**The daily smoke check (minutes).** Glance at the [metrics dashboard](TODO-path/metrics) in the Phoenix UI under your project's Metrics tab for anything on fire — latency quantiles spiking, error counts climbing, costs drifting. Then run a quick error query. The dashboard is a smoke detector, not a review process: it tells you *that* something changed, never *why*. The why always lives in individual traces.

**The weekly error-analysis session (an hour or two).** Pull a deliberately mixed sample of traces and read them with open-ended notes. This is where actual improvement comes from, and the second half of this guide is devoted to it.

Teams that only do the daily check end up dashboard-watching. Teams that only do deep dives get surprised by fires. You need both, and they feed each other: smoke-check anomalies tell you where to oversample in the weekly review.

## The Daily Check: Finding Hard Failures

Hard failures are queryable with zero setup — Phoenix records span status, exception events, and error messages on every trace. Your first triage move, straight from the terminal (see [Retrieve Traces via CLI](TODO-path/retrieve-traces-via-cli) for the full command reference):

```bash
# Find traces with errors in the last hour
px trace list --last-n-minutes 60 --format raw --no-progress | jq '.[] | select(.status == "ERROR")'

# List all errored spans
px span list --status-code ERROR --limit 20

# Drill down into a specific trace
px trace get <trace-id> --format raw | jq '.spans[] | select(.status_code != "OK")'
```

Two principles for reading error traces:

**Find the first failure, not the loudest one.** In multi-step agent traces, upstream errors cascade — a malformed tool output at step 2 produces a confused plan at step 4 and a wrong answer at step 9. Note the *first* point where the trace went wrong; fixing it usually dissolves the downstream symptoms.

**Group before you fix.** A hundred error traces might be three actual problems. Resist debugging trace-by-trace; collect, categorize, count. The weekly session below is where that happens.

## Finding Performance Outliers

Latency and cost problems rarely announce themselves in averages. A healthy p50 can hide a p99 where agents loop, contexts balloon, and users give up. Sort by extremes and read **both tails**:

* **Slow tail:** retry storms, agents stuck in tool-call loops, oversized retrieved contexts, sequential calls that should be parallel.
* **Fast tail:** short-circuits and silent failures — empty retrieval results, guardrails firing when they shouldn't, models returning refusals or stubs.

```bash
# Find the slowest traces (descending by duration)
px trace list --format raw --no-progress | jq 'sort_by(-.duration) | .[0:10]'

# Find suspiciously fast traces (ascending by duration)
px trace list --format raw --no-progress | jq 'sort_by(.duration) | .[0:10]'
```

```python
perf_query = SpanQuery().where("span_kind == 'LLM'").select(

p95_duration = df['duration'].quantile(0.95)
p99_duration = df['duration'].quantile(0.99)
p5_duration = df['duration'].quantile(0.05)

# Find slow tail (potential loops, retries)
slow_tail = df[df['duration'] > p95_duration].sort_values('duration', ascending=False)
print(f"Slow tail (>{p95_duration}ms): {len(slow_tail)} spans")

# Find fast tail (potential silent failures)
fast_tail = df[df['duration'] < p5_duration].sort_values('duration')
print(f"Fast tail (<{p5_duration}ms): {len(fast_tail)} spans")

# High token usage (cost outliers)
p95_tokens = df['total_tokens'].quantile(0.95)
high_cost = df[df['total_tokens'] > p95_tokens]
print(f"High token usage (>{p95_tokens}): {len(high_cost)} spans")
```

For *regressions* specifically, the unit of comparison is the change: compare the latency or token distribution before and after a deploy, prompt revision, or model swap — not against an absolute threshold. And whatever cutoff you use for "outlier" — p95, one standard deviation from the median — treat it as a starting point, not a finding. The threshold's only job is to hand you a tail worth reading.

## Finding Quality Issues with Evaluations

Evaluation results attached to traces let you query for soft failures the way you query for exceptions. Treat eval results as exploration clues, not verdicts:

* **Review both ends of the distribution.** Failing labels show you problems; passing labels show you whether your evaluator is calibrated. If you only ever read failures, you'll never notice your judge passing garbage.
* **Prefer binary and categorical judgments to numeric scales.** "Did the assistant hand off to a human when it should have: yes/no" is actionable. "Helpfulness: 4.2" is not. This isn't just taste — Arize research testing LLM judges on continuous ranges found numeric score evals unreliable, with small prompt changes producing wildly different results, and a 2025 re-test on newer models confirmed that binary and categorical judgments remain the most stable ([original study](https://arize.com/blog-course/numeric-evals-for-llm-as-a-judge/), [follow-up](https://arize.com/blog/testing-binary-vs-score-llm-evals-on-the-latest-models/)). Phoenix's built-in evaluators emit categorical labels for exactly this reason.
* **A 100% pass rate is a warning sign.** It usually means your evals aren't stressing the system. A 70% pass rate often indicates an eval that's actually testing something.

A practical consequence of label-based evals: **filter on labels, not score thresholds.** Score direction varies by evaluator (for Phoenix's hallucination evaluator, a *higher* score means *factual* — direction is "maximize"), so label filters are both more readable and harder to get backwards.

```bash
# List spans with their evaluation results
px span list --include-annotations --limit 20 --format raw --no-progress | \
  jq '.[] | select(.annotations != null) | {span_id: .context.span_id, name, annotations}'

# Filter traces by annotation label (requires jq post-processing)
px trace list --include-annotations --format raw --no-progress | \
  jq '.[] | select(.annotations[] | select(.name == "correctness" and .label == "incorrect"))'
```

## Session Signals: Multi-Turn Failures

Single-trace filters miss a whole class of problems that only exist across turns: the user who rephrases three times, the conversation that derails halfway, the agent that loses context it had two turns ago. Phoenix sessions group traces into conversations so you can read the interaction the way the user experienced it.

Session-level signals worth querying for:

* Sessions containing any error trace
* Unusually long sessions (turn count as an outlier metric — often a user fighting the system)
* Sessions with negative feedback on any turn

When reading a session, the first-failure principle matters even more: find the earliest turn where things went wrong, because every turn after it is contaminated context.

```bash
# List all sessions
px session list --limit 20

# Get a specific session with all its traces
px session get <session-id> --format raw | jq '.traces[] | {trace_id, start_time, status}'

# Find spans for a specific session
px span list --attribute session.id:my-session-123 --limit 50
```

## The Weekly Session: Finding Failure Clusters

Individual bad traces are anecdotes. Clusters are priorities. The method comes from qualitative research, and it works:

1. **Open coding.** Pull a sample of traces and write free-form notes on anything wrong with each one. Don't pre-define categories — that's how you miss the failure modes unique to your application.
2. **Axial coding.** Group the notes into a failure taxonomy: distinct, named categories. An LLM is genuinely useful for this bucketing step.
3. **Count.** Tally failures per category. This single step converts vibes into a prioritized roadmap. In one of Hamel Husain's client engagements, three categories accounted for over 60% of all problems.
4. **Iterate to saturation.** Keep sampling until new traces stop revealing new categories. Rule of thumb: review at least 100 traces to start; once ~20 consecutive traces turn up nothing new, you've saturated.

Random sampling gets inefficient once the obvious failures are fixed — most random traces are fine. Bias your sample toward signal:

* **Stratified sampling.** Group traces by the dimensions you care about — user segment, model, feature, query category, which tools were called, session turn count — and sample from each group, so a high-volume happy path doesn't drown out a broken niche. For agent applications, tools-called is often the most failure-correlated dimension you have, and it costs nothing: it's already an attribute on your spans. In Phoenix, the attributes you attach at instrumentation time (user IDs, session IDs, custom metadata) are exactly what makes this possible: stratification is an instrumentation decision before it's a query.
* **Embedding clustering (advanced, optional).** Embed your inputs and cluster them to reveal natural groupings, then sample from every cluster. How you weight the samples depends on your goal: weight proportionally to cluster size when you want representative failure-rate estimates, or oversample the small clusters when you're hunting for failure modes you haven't seen yet — the first measures, the second discovers. There's no "correct" clustering; it's an exploration technique for surfacing patterns you'd miss manually. This technique requires more setup but can surface latent failure modes you wouldn't find with manual filtering.

**Stratified sampling** by custom attributes:

```bash
# Sample spans from different user segments
```

```python
from phoenix.client import Client
from phoenix.trace.dsl import SpanQuery
import numpy as np
from sklearn.cluster import KMeans
from sklearn.decomposition import PCA
from openai import OpenAI  # or your embedding provider

client = Client()
openai_client = OpenAI()

# Export spans with inputs and eval results
query = SpanQuery().select(
    input="input.value",
    output="output.value",
    status="status_code",
    correctness_label="evals['correctness'].label",
)
df = client.spans.get_spans_dataframe(query=query)

# Embed the inputs
def get_embedding(text: str) -> list[float]:
    if not text or not isinstance(text, str):
        return [0.0] * 1536  # Zero vector for missing inputs
    response = openai_client.embeddings.create(
        model="text-embedding-3-small",
        input=text[:8000]  # Truncate long inputs
    )
    return response.data[0].embedding

df['embedding'] = df['input'].apply(get_embedding)
embeddings = np.array(df['embedding'].tolist())

# Cluster the embeddings (try 5-10 clusters to start)
n_clusters = 8
kmeans = KMeans(n_clusters=n_clusters, random_state=42)
df['cluster'] = kmeans.fit_predict(embeddings)

# Find clusters with elevated failure rates
df['is_error'] = df['status'] == 'ERROR'
df['is_incorrect'] = df['correctness_label'] == 'incorrect'
cluster_stats = df.groupby('cluster').agg(
    error_rate=('is_error', 'mean'),
    incorrect_rate=('is_incorrect', 'mean'),
    size=('input', 'count'),
)

print(cluster_stats.sort_values('error_rate', ascending=False))

# Oversample from high-error clusters for review
high_error_clusters = cluster_stats[cluster_stats['error_rate'] > 0.2].index
high_error_sample = df[df['cluster'].isin(high_error_clusters)].sample(n=min(50, len(df)))

# Also sample from small clusters (edge cases)
small_clusters = cluster_stats[cluster_stats['size'] < cluster_stats['size'].quantile(0.25)].index
edge_case_sample = df[df['cluster'].isin(small_clusters)].sample(n=min(20, len(df)))

print(f"\nHigh-error sample: {len(high_error_sample)} spans")
print(f"Edge-case sample: {len(edge_case_sample)} spans")

# Optional: visualize with PCA
pca = PCA(n_components=2)
coords = pca.fit_transform(embeddings)
df['x'] = coords[:, 0]
df['y'] = coords[:, 1]
# Now plot df[['x', 'y', 'cluster', 'status']] with your favorite plotting library
```

## Putting It Together: The Triage Loop

The full weekly loop, end to end:

1. **Pull a mixed sample.** All hard failures since last review, both latency tails, the lowest (and a few highest) eval scores, sessions with negative feedback or high turn counts, plus a stratified random slice so you don't develop tunnel vision.
2. **Annotate, open-coding style.** Free-form notes, first failure first. Phoenix annotations keep notes attached to the trace where the next reviewer can find them.
3. **Update the taxonomy and re-count.** New failure modes get new categories; recurring ones get their tallies bumped.
4. **Prioritize by frequency × impact** and fix from the top.
5. **Convert each confirmed failure mode into an evaluator** — a code assertion where possible, an LLM-as-judge with validated binary outputs where not. This is the flywheel: every review cycle makes the next cycle's automated filtering smarter.

**Where agents fit.** The filtering and sampling steps above are mechanical enough to delegate to a coding agent once your sampling logic is settled. More interestingly, agent-as-judge review over *full trajectories* adds a recall layer that scalar filters miss: an agent can assess whether a complex tool call or generated shell command was actually correct — something no human reviewer can verify at scale — and can contextualize the smoke signals your filters raise. The division of labor: agents triage and judge what humans can't check at volume; humans keep what agents can't be trusted with — validating the judge's calibration, naming new failure categories, and deciding what matters. The thesis doesn't change: agent review is one more filter whose escalations a human reads.

The scripts below are **starting-point templates** — adapt the sampling logic, thresholds, and time windows to match your application's failure modes and review capacity.

```python
p5_duration = spans_df['duration'].quantile(0.05)
fast_tail = spans_df[spans_df['duration'] < p5_duration]

    errors_df.head(20),     # Top 20 errors
```

## Next Steps

* Convert your top failure categories into automated evaluators and run them as online evals
* Set up annotation queues so domain experts — not just engineers — review edge cases; the people best positioned to judge quality often write no code
* Track failure-category counts over time: a category trending up is a regression even if no individual eval fires
* Revisit the full loop after every significant prompt, model, or retrieval change

</Warning>

PXI (pronounced *"pixie"*) is an agent built into Phoenix, modeled after a
coding agent but pointed at your observability data instead of a code editor. It
sees what you are looking at and acts on your behalf — driving the product,
investigating failures, and iterating on prompts and traces in context.

{/* VIDEO PLACEHOLDER: end-to-end demo of PXI driving Phoenix — opening a failing trace, proposing a root cause, then jumping to the playground to fix the prompt. */}

## What it does

- **Drives the product** — navigates, filters, and pivots through your Phoenix
  data the same way you would, only faster.
- **Investigates failures** — walks failing traces and proposes root causes
  instead of leaving you to grep through spans.
- **Iterates on prompts** — reads, edits, and tests playground prompts, with
  every change shown as a diff you approve.
- **Reasons over your data** — a sandboxed runtime lets PXI query your Phoenix
  instance to answer questions evals and dashboards cannot.
- **Knows the product** — Phoenix's own documentation is wired in as a
  first-class source, so answers are grounded rather than guessed.

Because PXI is grounded in your current context, what it can do adapts to the
page you are on — its capabilities on a trace differ from those in the prompt
playground.

## Skills

Skills are PXI's superpower. A **skill** is a reusable, multi-step playbook that
teaches PXI how to do one Phoenix workflow *well* — what to look at, in what
order, and what good output looks like. PXI loads the matching skill on demand
and follows a curated procedure built by the Phoenix team.

The library is **under active development** and grows every release — track
progress on the
[Phoenix roadmap milestone](https://github.com/Arize-ai/phoenix/milestone/13).

PXI ships with additional skills today, including **span annotation** and
**LLM-as-a-judge evaluator authoring**. Because skills are context-aware, PXI
surfaces the right one for the page and task you are on.

## You stay in control

PXI is built so that agent assistance is something you **opt into, not something
that happens to you.** It can be turned off completely, runs under an explicit
permission model, and only reaches the internet when you let it.

<Frame caption="Settings → Assistant — system, personal, and web-access controls in one place.">
  ![PXI assistant settings](https://storage.googleapis.com/arize-phoenix-assets/assets/images/pxi/assistant_settings.png)
</Frame>

- **Turn it fully off.** PXI is the only agent in its class you can switch *all
  the way off* — there is no always-on mode you cannot escape. Disable it per
  deployment (`PHOENIX_DISABLE_AGENT_ASSISTANT=true`), per instance
  (**Settings → Assistant → System settings**), or per browser
  (**Settings → Assistant → Personal settings → Use assistant**).
- **Nothing changes without your approval.** Any state-changing action — editing
  a prompt, saving a prompt, annotating spans — is gated by an edit-approval
  mode you pick from the chat input (or cycle with `Ctrl+T`). Read-only actions
  run freely.
- **Add web grounding when you want it.** Toggle **web access** with the globe
  button in the chat input to let PXI consult the live internet for additional
  grounding. The toggle is per session and only appears when an administrator
  allows it; leave it off to keep the session entirely inside your Phoenix
  instance.

| Edit-approval mode | Behavior |
| --- | --- |
| **Manual Approval** *(default)* | PXI proposes the change as a reviewable diff and waits. Nothing is applied until you click **Accept**. |
| **Bypass Approval** | Edits are applied without asking. The selector shows a warning treatment while active. |

## Setup

PXI needs a model to talk to. Configure credentials for at least one provider
via environment variables or Phoenix secrets (`OPENAI_API_KEY`,
`ANTHROPIC_API_KEY`, `GEMINI_API_KEY`, AWS Bedrock credentials, or a custom
provider under **Settings → Models**).

The first time you open PXI you are shown a consent gate to review the assistant
session trace settings. Acknowledging it enables the chat surface for that
browser; it does not override system settings.

<Accordion title="Pick a recommended model">
PXI relies heavily on **tool calling** — almost every action it takes is a tool
call. Models that are weak at tool use produce broken sessions even if they
handle free-form chat well. Pick one of these validated models unless you have a
specific reason not to:

- **Anthropic** — `claude-opus-4-8`, `claude-opus-4-6`, `claude-sonnet-4-6`
- **OpenAI** — `gpt-5.5`, `gpt-5.4`, `gpt-5.4-mini`
- **Google** — `gemini-3.1-pro-preview`, `gemini-3.5-flash`

Other built-in or custom-provider models can be selected from the model menu,
but they are untested with PXI and may fail to invoke tools correctly.
</Accordion>

## How it works

PXI is split between the Phoenix server, which owns everything the model sees
(tool definitions, system prompt, skills, capability guidance), and the browser,
which executes tool calls that touch the page. Capabilities are gated by
**context** — PXI only advertises a tool when the required Phoenix UI context is
present, so it never offers an action that cannot succeed on your current page.

<AccordionGroup>
  <Accordion title="Architecture">

  ```mermaid
  flowchart LR
      User((User))

      subgraph Browser["Browser"]
          direction TB
          ChatUI["Chat UI"]
          Bash["bash sandbox"]
          PhoenixGQL["phoenix-gql CLI"]
          ChatUI ~~~ Bash
          Bash --> PhoenixGQL
      end

      subgraph Server["Phoenix Server"]
          direction TB
          GraphQL["GraphQL API"]
          Agent["PXI Agent"]
          Tools["Tools"]
          Skills["Skills"]
          MCP["MCP"]
          Agent --> Tools
          Agent --> Skills
          Agent --> MCP
      end

      LLM["LLM Provider"]
      Docs["Mintlify Docs MCP"]

      User <--> ChatUI
      ChatUI <--> Agent
      PhoenixGQL --> GraphQL
      Agent <--> LLM
      MCP --> Docs
  ```

  The browser owns the chat UI and a sandboxed **bash** environment. Inside it,
  PXI uses the **phoenix-gql** CLI to query the Phoenix **GraphQL API** — the
  same authenticated endpoint a logged-in user hits. The server hosts the agent
  and its model-facing surface (**tools**, **skills**, an **MCP** client) and
  calls your LLM provider with your API key. When external resources are
  allowed, the server also reaches the Mintlify-hosted Phoenix docs MCP.
  </Accordion>

  <Accordion title="Extending PXI">
  Because PXI is built like a coding agent, its capabilities grow by adding to a
  registry, not by retraining a model. There are three extension points, all
  owned by the server as the single source of truth:

  - **Tools** — the actions PXI can take, in three flavors: *external* (always
    run in the browser), *contextual client* (gated on the page you are
    viewing), and *contextual server* (read or compute over data the server has
    authenticated access to).
  - **Skills** — the playbooks above. Adding one is as simple as authoring a
    `SKILL.md`; PXI discovers and loads it on demand. This is how new workflows
    ship without changing the core agent.
  - **MCP** — PXI is an MCP client. Today it connects to the Phoenix docs server
    for grounded product knowledge; the same path reaches new context sources
    over time.

  Every new tool or skill inherits the same guarantees automatically: it is
  context-gated, runs under the permission model, and is captured in traces when
  recording is on.
  </Accordion>

  <Accordion title="Runs entirely on your Phoenix">
  PXI runs **inside the Phoenix process you are already running**:

  - Tool calls execute against your Phoenix server and your data — no separate
    Arize service is involved.
  - The LLM is **your model provider**, called with **your API key**. Arize is
    not in the request path.
  - Documentation lookups go to the **Mintlify-hosted Phoenix docs MCP server**
    when external resources are allowed — the same public docs you can read in a
    browser, serving docs only.
  - Remote trace export happens **only if every gate is enabled**: a remote
    collector is configured, an administrator allows export in system settings,
    and the user enables it in personal settings.
  </Accordion>
</AccordionGroup>

## Privacy, safety & configuration

<AccordionGroup>
  <Accordion title="Trace recording & observability">
  PXI can capture conversations as Phoenix traces, controlled by both system
  settings and per-browser preferences. From **Settings → Assistant**,
  administrators can turn assistant access on or off for everyone, allow users to
  save session traces locally, and allow export to a configured remote collector;
  each user can show or hide the assistant and opt into local or remote trace
  recording when allowed.

  By default the system settings allow **neither** local persistence nor remote
  export. Local traces are written to the `assistant_agent` project (override
  with `PHOENIX_AGENTS_ASSISTANT_PROJECT_NAME`). When recording is enabled, tool
  inputs and outputs are recorded on the corresponding spans, so you can audit
  what PXI did and evaluate it like any other agent in Phoenix.
  </Accordion>

  <Accordion title="Safety limits">
  - **Verify before you act.** PXI can apply filters, edit prompts, and run
    bash. Review proposed changes — especially prompt edits — before accepting.
  - **Don't point it at sensitive production data without controls.** PXI sees
    whatever the signed-in user can see; use scoped or staging projects while it
    is in beta.
  - **Treat outputs as suggestions.** PXI hallucinates, especially on long
    traces or unfamiliar frameworks.
  </Accordion>

  <Accordion title="Configuration reference">
  | Environment variable | Effect |
  | --- | --- |
  | `PHOENIX_DISABLE_AGENT_ASSISTANT=true` | Disable PXI for the whole deployment (requires restart). |
  | `PHOENIX_ALLOW_EXTERNAL_RESOURCES=false` | Disable web access *and* external docs lookups. |
  | `PHOENIX_AGENTS_DISABLE_WEB_ACCESS=true` | Disable PXI's web search/fetch tools while leaving other external resources available. |
  | `PHOENIX_AGENTS_COLLECTOR_ENDPOINT` | Remote collector endpoint for assistant trace export. |
  | `PHOENIX_AGENTS_COLLECTOR_API_KEY` | API key for the remote collector, if required. |
  | `PHOENIX_AGENTS_ASSISTANT_PROJECT_NAME` | Project name for locally recorded assistant traces (default `assistant_agent`). |

  The old `PHOENIX_DANGEROUSLY_ENABLE_AGENTS` opt-in flag is no longer used.
  </Accordion>
</AccordionGroup>

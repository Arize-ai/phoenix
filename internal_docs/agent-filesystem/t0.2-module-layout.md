# T0.2 Module Layout and Naming

This note fixes the initial frontend module layout for the agent `bash` runtime and filesystem adapters.

## Placement decisions

- Keep chat UI components in `app/src/components/agent/`.
- Add client tool execution code under a new domain folder: `app/src/agent/`.
- Keep React wiring thin in `app/src/components/agent/Chat.tsx` and move runtime/adapter logic into plain TypeScript modules under `app/src/agent/`.
- Reuse the existing agent session identity from `app/src/contexts/AgentContext.tsx` and `app/src/store/agentStore.ts` so the runtime can stay scoped per chat session.

## Initial file layout

```text
app/src/agent/
  tools/
    bash/
      bashToolSchema.ts
      bashToolRuntime.ts
      bashToolSessionRegistry.ts
      bashToolTypes.ts
      index.ts
  context/
    adapters/
      types.ts
      composeAdapters.ts
      pageContextAdapter.ts
      index.ts
    filesystem/
      manifest.ts
      contextMetadata.ts
      indexFiles.ts
      schemaFiles.ts
      pathConstants.ts
      index.ts
```

## Naming rules

- Use `bashTool*` for the client-executed `bash` runtime modules so they stay easy to grep and distinct from generic shell helpers.
- Use `ContextAdapter` names for adapter contracts and `*Adapter` suffixes for implementations.
- Use `pathConstants.ts` for the stable `/phoenix` skeleton and path helpers.
- Use focused helper filenames for generated filesystem artifacts instead of one large catch-all utility module.

## Ownership by task

- T1.2: `app/src/agent/tools/bash/bashToolSchema.ts`
- T1.3: `app/src/agent/tools/bash/bashToolRuntime.ts`, `app/src/agent/tools/bash/bashToolSessionRegistry.ts`
- T2.1-T2.4: `app/src/agent/tools/bash/*`
- T3.1: `app/src/agent/context/adapters/types.ts`
- T3.2: `app/src/agent/context/filesystem/*`
- T3.3: `app/src/agent/context/adapters/composeAdapters.ts`
- T4.1: `app/src/agent/context/filesystem/pathConstants.ts`
- T5.1: `app/src/agent/context/adapters/pageContextAdapter.ts`

## Why this layout

- `app/src/components/agent/` already contains the view layer, so keeping new runtime logic outside that folder avoids mixing React rendering with browser-runtime and filesystem concerns.
- `app/src/agent/` is broad enough to hold future non-UI agent modules without forcing them into `components/`, `contexts/`, or `utils/`.
- Splitting `tools/bash` from `context/` keeps the execution runtime separate from Phoenix-context materialization, which matches the task graph and should make follow-up verification easier.

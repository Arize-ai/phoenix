# T0.1 Frontend Hybrid Tool Contract

This note locks the v1 contract for client-executed tools before the `bash` runtime work starts.

## Decision: execution location stays frontend-local in v1

- Do not add a `location: "client" | "server"` field to the `/chat` request payload yet.
- In v1, any tool definition the frontend includes in the chat request is treated as a client-executed tool.
- The frontend will keep the execution-location decision in a local registry/module and map tool names to client handlers.
- The backend remains transport-only for these tools: it forwards the schema to the model and streams tool-call parts back to the UI.

This keeps the wire contract aligned with the existing backend `FrontendTool` model in `src/phoenix/server/api/routers/data_stream_protocol.py`.

## Request contract

- `POST /chat` already accepts a `tools` array in the JSON body via `_VercelRequest.tools` in `src/phoenix/server/api/routers/data_stream_protocol.py`.
- The agent chat client should send the `bash` tool definition on every turn, alongside `messages` and optional `system`.
- The initial v1 tool schema should stay minimal and stable:

```json
{
  "name": "bash",
  "description": "Run a shell command in the browser virtual filesystem",
  "parameters": {
    "type": "object",
    "properties": {
      "command": {
        "type": "string",
        "description": "Shell command to execute"
      }
    },
    "required": ["command"],
    "additionalProperties": false
  }
}
```

## Stream/UI contract

- The backend already emits dynamic tool-call chunks for tool invocations in `src/phoenix/server/api/routers/data_stream_protocol.py`.
- Tool calls are streamed with `ToolInputStartChunk`, `ToolInputDeltaChunk`, and `ToolInputAvailableChunk` and marked `dynamic=True`.
- This is the AI SDK path that the frontend can handle with `onToolCall`, answer with `addToolOutput`, and resume with `sendAutomaticallyWhen: lastAssistantMessageIsCompleteWithToolCalls`.
- The UI message stream therefore already has the right shape to represent client-rendered tool activity; follow-up work only needs to render those parts and wire execution.

## Frontend touchpoints for follow-up tasks

- `app/src/components/agent/Chat.tsx`: send tool definitions with each request and handle client tool execution.
- `app/src/components/agent/ChatMessage.tsx`: render tool-call/tool-result parts.
- `src/phoenix/server/api/routers/data_stream_protocol.py`: keep accepting `tools` and streaming dynamic tool-call parts.
